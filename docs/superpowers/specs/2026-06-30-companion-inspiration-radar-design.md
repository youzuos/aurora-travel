# 旅行 Agent 灵感雷达与计划精细化设计

日期：2026-06-30

## 背景

当前 Aurora 已经实现旅行 Agent：小动物可以在城市中旅行、聊天、发照片、发语音，并且用户可以指定它去某个城市。年度计划本身也已有时间推进能力，计划会随着 `TimeWarp` 从年初、出发前 3 月、出发前 1 月到 T-14 逐步变清晰。

下一步要把这两个能力连接起来：让小动物不只是聊天入口，而是旅行计划逐步精细化的探索来源。小动物在城市里旅行时会发回见闻；用户如果对某个地方感兴趣，可以把它加入本次旅行清单或年度愿望清单。用户也可以派小动物去任意城市、国家、地区或模糊旅行偏好；如果不输入，则小动物随机出发。

## 目标

1. 让用户通过小动物见闻发现新目的地，而不是只通过传统表单填写愿望。
2. 让“随着时间推进，计划逐步精细化”变得可感知：小动物在不同时间阶段补充更具体的旅行信息。
3. 输入界面避免传统聊天框形态，改为更产品化的“旅行灵感雷达 / 派小动物去看看”。
4. 支持自由输入，不限制用户只能选固定选项。
5. 支持当前城市附近推荐和地理距离计算。
6. 如果 IP 位置信号容易获得，则把它作为“随机出发”的轻量兜底，不作为强定位能力。

## 非目标

本阶段不实现以下内容：

1. 不做精确用户定位。
2. 不做用户当前位置授权。
3. 不做依赖第三方网络服务的 IP 定位。
4. 不依赖 Gemini 来决定地理距离或附近城市，避免不稳定。
5. 不默认实时生成旅行图片，仍使用城市照片池。
6. 不替换现有年度计划生成器，只增强愿望来源和计划精细化体验。

## IP 位置信号边界

如果部署环境能直接提供请求头中的粗略地区信息，则可以读取这些字段作为非阻塞增强：

- `x-vercel-ip-country`
- `x-vercel-ip-city`
- `cf-ipcountry`
- `x-country-code`
- `x-city`

该能力只用于空输入随机出发时的轻量偏好。例如检测到用户可能在日本，则随机出发可以优先从日本城市中选一个。界面不展示“我知道你在某地”，也不把 IP 结果作为事实声明。

如果没有这些请求头，或者请求头无法匹配本地城市库，则静默退回普通随机出发。核心功能不能依赖 IP。

## 推荐方案

采用“本地城市库 + 经纬度距离计算 + 小动物见闻卡片”的方案。

每个可探索城市在本地数据中维护：

- 城市 ID
- 中文名 / 英文名
- 国家中文名 / 英文名
- 经纬度
- 标签，如海边、美食、雪山、樱花、极光、小众、古城
- 照片池
- 见闻素材
- 可选季节信息

用户输入时，系统先在本地城市库中匹配城市、国家或标签。用户点击“继续附近探索”时，系统基于小动物当前城市的经纬度计算距离，选择最近且未刚刚访问过的城市。用户不输入时，从城市库中随机选择一个城市。

## 用户体验设计

### 入口形态

入口不做成传统聊天窗口，而是一个“旅行灵感雷达”模块，可以出现在以下位置：

1. 无计划状态下的首页，用于启动年度旅行灵感。
2. 已有计划后的年度视图，用于继续发现可加入的地点。
3. 小动物聊天面板内作为一个独立操作区，但不是普通消息输入框。

建议文案：

```text
想让小动物先去哪里看看？
[ 输入任意城市、国家、地区或旅行偏好 ]

例如：东京 / 日本 / 巴黎 / 冰岛 / 海边 / 雪山 / 小众古城 / 适合吃东西的地方
```

下方可以有快捷 chips，但只作为灵感提示，不限制输入范围：

- 海边
- 美食
- 樱花
- 极光
- 小众
- 古城
- 不输入，随机出发

另外提供一个明确按钮：

- 继续附近探索

这里的“附近”只指小动物当前所在城市附近，不指用户所在地。

### 用户输入规则

用户可以输入任意文本。

处理规则：

1. 输入具体城市，如“巴黎”：小动物直接去巴黎。
2. 输入国家，如“日本”：从该国家的城市中选择一个。
3. 输入地区或模糊偏好，如“海边”“雪山”“适合吃东西的地方”：按城市标签匹配。
4. 不输入：随机选择一个城市；如果有可用的粗略 IP 国家信号，可以优先从该国家的城市中选择。
5. 点击“继续附近探索”：基于小动物当前城市计算最近城市。

如果匹配不到结果：

- 显示友好提示：“我还没找到这个地方，要不要先随机出发？”
- 提供按钮：随机出发、换个关键词、继续当前城市附近探索。

### 见闻卡片

小动物到达或探索到一个城市后，发回一张“旅行见闻卡片”。

卡片内容：

- 小动物头像和状态
- 城市 / 国家
- 与上一个城市的距离，若是附近探索触发
- 一段拟人化见闻
- 城市照片
- 标签，如美食、海边、古城
- 当前计划成熟度阶段

卡片操作：

- 加入本次旅行清单
- 加入年度愿望
- 继续附近探索
- 换个方向
- 让它去指定地方

“加入本次旅行清单”用于已有计划时，将地点加入当前选中的 trip 或计划候选清单。“加入年度愿望”用于无计划或未选中 trip 时，将地点作为 wishlist item 进入年度计划生成流程。

## 计划精细化设计

小动物见闻应随着 `TimeWarp` 阶段逐步变具体。

### 年初

信息偏灵感层：

- 这里为什么值得去
- 适合什么类型旅行
- 大致季节
- 小动物看到的第一印象

示例：

```text
我在巴黎河边走了一圈，觉得这里适合放进一场慢一点的城市散步旅行。
```

### 出发前 3 月

信息偏规划层：

- 推荐月份或季节
- 大致预算感知
- 是否适合与附近城市组合
- 旅行天数建议

示例：

```text
如果把巴黎放在春末，3 到 5 天会比较舒服。我还发现它可以和布鲁塞尔或阿姆斯特丹连成一条线。
```

### 出发前 1 月

信息偏行程层：

- 具体街区
- 景点或体验
- 餐食方向
- 照片点

示例：

```text
我今天把路线缩小到塞纳河两岸，傍晚可以从桥上看光慢慢落到水面。
```

### T-14

信息偏临近提醒：

- 天气提醒
- 交通或预约提醒
- 备选方案
- 打包建议

示例：

```text
我看了临近天气，傍晚风会有点凉。围巾和轻外套要放进包里，雨天可以把室外散步换成博物馆。
```

## 数据模型设计

### 新增探索城市字段

在目的地或 companion location 数据中增加：

```ts
type Coordinates = {
  lat: number;
  lng: number;
};

type ExplorationCity = {
  id: string;
  cityZh: string;
  cityEn: string;
  countryZh: string;
  countryEn: string;
  coordinates: Coordinates;
  tags: string[];
  photos: CityPhoto[];
  snippets: {
    inspiration: string[];
    planning: string[];
    itinerary: string[];
    finalCheck: string[];
  };
};
```

可优先复用当前 `data/destinations.ts` 和 `data/companion.ts` 中已有地点，再补充经纬度和标签。

### 新增见闻卡片

```ts
type CompanionFinding = {
  id: string;
  cityId: string;
  source: "direct-input" | "country-match" | "tag-match" | "random" | "nearby";
  distanceKm?: number;
  textZh: string;
  textEn: string;
  photo?: CityPhoto;
  tags: string[];
  maturityStage: WarpStop;
  createdAt: number;
  addedToWishlist: boolean;
};
```

### 加入清单

当用户点击加入清单时，将见闻转换为 `WishlistItem`：

```ts
{
  id: `companion-${finding.id}`,
  label: cityName,
  priorityLabel: "想去",
  priorityScore: 2,
  source: "companion"
}
```

已有计划状态下，如果后续支持“本次旅行清单”，可以先进入当前 trip 的候选清单；本阶段若没有 trip-level 清单模型，则先加入年度 wishlist，避免扩大数据结构。

## 核心逻辑

### 目的地解析

解析优先级：

1. 城市名精确匹配。
2. 国家名匹配。
3. 标签匹配。
4. 随机城市。

国家匹配时，如果小动物当前城市所在国家与输入国家相同，优先选择该国家中未刚刚访问过的城市；否则随机选择该国家的城市。

标签匹配时，可根据标签命中数量排序。

### 附近探索

附近探索只使用小动物当前城市。

算法：

1. 读取当前城市经纬度。
2. 排除当前城市。
3. 排除最近若干次已经访问过的城市。
4. 使用 Haversine 公式计算距离。
5. 选择最近城市，或在最近 3 个城市中轻微随机，避免每次完全一样。

距离展示：

- 1000km 以下显示整数公里。
- 1000km 以上显示约数，如“约 1,200km”。

### 随机出发

不输入时随机选择城市。

随机策略：

- 避免选择当前城市。
- 避免连续重复最近访问城市。
- 可在城市标签间做轻微均衡，避免总是热门城市。

## UI 组件设计

### `CompanionInspirationRadar`

职责：

- 展示开放式目的地输入器。
- 展示快捷 chips。
- 提供随机出发和附近探索按钮。
- 调用探索逻辑生成 finding。

状态：

- 输入文本
- 正在探索
- 错误提示
- 最近生成的见闻卡片

### `CompanionFindingCard`

职责：

- 展示城市、照片、见闻、标签、距离。
- 提供加入清单、继续附近探索、换个方向、指定地方操作。

### `useCompanionExploration`

职责：

- 封装目的地解析、附近计算、随机选择、finding 生成。
- 保持 UI 组件简单。

可以先不做成 hook，如果现有代码更适合放在 `lib/companion.ts`，则优先保持现有项目风格。

## 与现有功能的关系

### 与 `CompanionChat`

聊天面板继续保留，但灵感雷达不是传统聊天输入的替代品。它可以作为聊天面板顶部或独立 tab 出现，让用户“派小动物探索”。

### 与 `ChatOverlay`

当前年度计划输入是分步表单。灵感雷达不应把用户拉回问答式流程，而是把见闻结果转换为 `WishlistItem`，再交给已有年度计划流程。

### 与 `YearView`

无计划状态下，`YearView` 可以展示灵感雷达作为更自然的启动方式，而不是只显示“告诉我你这辈子想看的风景”。

已有计划状态下，`YearView` 可以展示小动物最近见闻，并让用户把地点加入清单，让计划随着时间推进变精细。

### 与 `TimeWarp`

`TimeWarp` 决定见闻卡片的具体程度。不同时间阶段使用不同 snippet 组，或在 LLM prompt 中传入 maturity stage 让 Gemini 改写。

本阶段推荐本地 snippet 为主，Gemini 仅增强文本，避免核心功能依赖网络。

## 错误处理

1. 用户输入为空：执行随机出发。
2. 输入无法匹配：提示换关键词或随机出发。
3. 城市缺少坐标：附近探索时跳过该城市。
4. 照片加载失败：沿用现有照片 fallback，显示文本见闻。
5. Gemini 不可用：使用本地见闻文案。
6. 用户重复加入同一城市：提示已在清单中，或把优先级提高。

## 测试计划

### 单元/回归测试

1. 输入具体城市能匹配正确城市。
2. 输入国家能匹配该国家下城市。
3. 输入标签能匹配带该标签城市。
4. 空输入会随机选择非当前城市。
5. 附近探索按距离返回最近城市。
6. 附近探索不会返回当前城市。
7. 最近访问城市会被排除。
8. finding 可以转换为 `WishlistItem`。
9. 不同 `WarpStop` 返回不同精细度文案。

### 浏览器测试

1. 无计划状态下可以通过灵感雷达随机出发。
2. 输入“巴黎”生成巴黎见闻卡片。
3. 输入“日本”生成日本城市见闻卡片。
4. 输入“海边”生成海边标签城市。
5. 点击“继续附近探索”显示距离和新城市。
6. 点击“加入年度愿望”后，计划输入流程中出现该地点。
7. 已有计划状态下，见闻卡片不会破坏 YearView 布局。

## 实施顺序

1. 扩展城市数据：补坐标、标签和分阶段见闻文案。
2. 实现目的地解析和距离计算。
3. 增加 finding 数据结构和转换 wishlist 的工具函数。
4. 实现 `CompanionFindingCard`。
5. 实现 `CompanionInspirationRadar`。
6. 接入无计划首页。
7. 接入已有计划年度视图。
8. 接入聊天面板中的非聊天式探索入口。
9. 补测试和浏览器验证。

## 验收标准

1. 用户可以输入任意城市、国家、地区或偏好。
2. 用户不输入时，小动物可以随机出发。
3. 用户可以点击当前城市附近探索，系统用经纬度计算附近城市。
4. 小动物会生成旅行见闻卡片。
5. 用户可以把感兴趣地点加入愿望/旅行清单。
6. 见闻内容会随 `TimeWarp` 阶段变得更具体。
7. 整个输入体验不表现为传统聊天框。
8. 无 Gemini key 时功能仍可用。
9. 回归测试和生产构建通过。
